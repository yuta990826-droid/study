1日目：【Stage 2】「1人2役」でプルリク・マージを完璧にする
実務の基本である「ブランチ作成 ➡️ プルリク ➡️ マージ」のサイクルを、脳に焼き付けます。

やること：

VS Codeのターミナルで git checkout -b feature/my-profile と打ち、新しい枝を作る。

新しいファイルを追加して、コミット ➡️ git push origin feature/my-profile でプッシュ。

ブラウザでGitHubを開き、自分でプルリクエストを作成。

自分で内容をチェックして「Merge pull request」を押して本番（main）に合流させる。

合格基準： カンペ（解説サイト）を見ずに、この一連の流れを3分以内にノーミスでできること。

📅 2日目：【Stage 3】最大の敵「コンフリクト（衝突）」をわざと起こして倒す
現場でみんなが一番恐れる「データの衝突」を、自分の手で人工的に作り出して解決します。

やること：

ブラウザ（GitHub）側で、README.md の1行目を「こんにちは」に書き換えて保存する。

次に、手元のVS Code側で、まだプル（ダウンロード）していない状態のまま、同じ1行目を「さようなら」に書き換えてコミットする。

そのままプッシュ（git push）を試みる ➡️ 絶対にエラー（Conflict）が起きます。

VS Codeの画面が赤や青になり「どちらの変更を残しますか？」と聞かれるので、指示に従って綺麗に修正し、再度コミット＆プッシュを完了させる。

合格基準： コンフリクトが起きたときに、パニックにならずに画面のコードを修正できるようになること。

📅 3日目：【Stage 3】タイムマシン機能（過去への逆行）を極める
「間違えて変なファイルをコミットしちゃった！」「2日前の状態に完全に巻き戻したい！」を叶える魔術を学びます。

やること：

git log --oneline と打ち、これまでの自分のセーブデータのID（ハッシュ値と呼ばれる英数字）を確認する。

以下の2つの呪文の違いを、ファイルを適当に書き換えながら実験する。

git reset --soft [セーブデータのID]：歴史は戻るけど、ファイルの中身は消さずに残す（やり直したい時）。

git reset --hard [セーブデータのID]：ファイルの中身も含めて、完全にその時点まで世界を巻き戻す（なかったことにしたい時）。

合格基準： 過去の任意のセーブポイントへ、自由自在に行き来できるようになること。

📅 4日目：【Stage 4】作業中のデータを「一時退避」させる達人になる
「ブランチAで作業中だけど、急にブランチBのバグを直してって言われた！でも今の作りかけのコードはまだコミットしたくない…」という実務あるあるを解決します。

やること：

ファイルを適当に書き換えた（作りかけの）状態で、git stash（ギット・スタッシュ）と打つ。

➡️ 画面から作りかけのコードが消え、最初の状態に戻ります（データはGitが一時的に預かってくれています）。

別の作業をしたあとに、git stash pop と打つ。

➡️ さっきの作りかけのコードが魔法のように画面に復活します。

合格基準： コミットせずに、作業状態を綺麗に保存してブランチ移動ができるようになること。

📅 5日目：【Stage 4】歴史の書き換え「Rebase（リベース）」
「Merge」とは異なる、もう1つの合流方法です。歴史の枝分かれをなくし、1本の綺麗な直線に整えるプロの技です。

やること：

git checkout -b feature/test で枝分かれさせた後、本番（main）側にも別のコミットを追加する。

その後、自分の枝で git rebase main と打つ。

git log --graph（または拡張機能のGitGraph）で、歴史の線がどう変わったか観察する。

合格基準： Merge（枝を残して合流）と Rebase（根元を移動して直列にする）の見た目の違いを理解すること。

📅 6日目：【プロレベル】「GitHub Actions」で全自動化の世界を体験する
あなたがプッシュした瞬間に、GitHubのクラウドロボットが自動で仕事を処理してくれる仕組みを作ります。

やること：

リポジトリの直下に .github というフォルダを作り、その中に workflows というフォルダを作る。

さらにその中に test.yml というファイルを作り、GitHub Actionsの簡単なサンプルコード（「プッシュされたらHello Worldと画面に出力する」など）をコピペして貼り付ける。

これをプッシュし、GitHubの「Actions」タブを開く。

ロボットが緑色のチェックマークをつけて自動で動いている感動を味わう。

📅 7日目：総仕上げ「実務シミュレーション」
最後に、1人で「完璧な開発の流れ」を再現します。

最終試練：

朝一番の git pull から始める。

機能ごとの丁寧な名前をつけた branch を作る。

意味のある単位で3回 commit を重ねる。

途中で1回 git stash を挟んで別ブランチを確認してみる。

本番へ push し、GitHubで綺麗な Pull Request を作成、自分で Merge する。
